GreatExpectations(v0)のProfilingとDataAsistantの特長#

GreatExpectations(v0)のデータプロファイリング機能の特長についてまとめます。

しかし昨年v1がリリースされ、従来(v0)のProfilingは将来的に使われなくなるようなので、ご参考程度にどうぞ。

https://docs.greatexpectations.io/docs/reference/learn/migration_guide/?expectation_config=v1_expectation_suite_config&expectation_suite_api=v1_expectation_suite_api&snowflake_config=v0_snowflake_config&data_context_config=v0_data_context_config&checkpoints_yaml=v1_checkpoints_yaml
「profilers: This feature has been removed in V1. Some form of profilers will be re-introduced in V1 at a later date.」

GreatExpectations(v0)のProfilerの特長#

GreatExpectationsではProfilerを利用しdata profilingをする。
主にサポートしているProfilerは次の3つである。

どのProfilerもprofiling結果はExpectation Suite(データの期待の集まり)を返す。

User Configurable Profiler#

基本的なdata pofilingをしてくれる、Profiler。

  • 対応データ
    • テーブル自体
    • カラム
      • 文字列
      • datetime
      • 数値
  • data profiling可能なExpectation
    • こちらのDocの末尾を参照

また「semantic_types」を利用することで、それぞれのカラムの型に合ったdata profilingへ分岐し実施することができる。

実行の仕方#

過去の記事を参考に https://467tn.com/post/content6/#profilingprofiler%E3%82%92%E4%BD%9C%E6%88%90%E3%81%97profiling%E3%81%97%E3%81%A6%E3%81%BF%E3%82%8B

JsonSchemaProfiler#

Jsonのスキーマ専用のdata profilingをしてくれる、Profiler。 ただ、ネストレベルの走査はしない。プロファイリングの内容は「User Configurable Profiler」と同じような内容をしている。

Rule-Based Profiler#

WARN：実験的な機能

前述の「User Configurable Profiler」「JsonSchemaProfiler」だと、プロファイリング内容が固定であり、柔軟なルールの設定ができない・データ設計に対する共通の設定ができないなどの問題がある。 そのような問題を解決するために作られたProfiler。

例えば#

例えば、以下のような主キーはidというカラム、外部キーは*_idという命名規則であるデータテーブルがあったとする。

– name table

idnameaddress_id
1Alice101
2Bob102
3Carol101

– address table

idname
101Tokyo

この時にプロファイリングとしては以下のようになって欲しいものだ。

  • idカラムは、主キーとして扱うためのプロファイリング(データがユニークである etc)
  • address_idカラムは、外部キーとして扱うためのプロファイリング(該当テーブルにidがある etc)

こういったデータの命名規則などの知識を活用して、適切なデータ品質ルール（Expectation）を自動的に適用できるようにするために 「Rule-Based Profiler」が作られた。

公式で用意されていた図解(わかりづらい)

profiler configの書き方#

まず、yamlベースでprofiler configを書く。

name: My Profiler
config_version: 1.0

variables: ・・・

rules:
  <rule名>:
      domain_builder: ・・・
      parameter_builder: <list>
      expectation_configuration_builder: <list>

各種キーについての詳しい説明：https://docs.greatexpectations.io/docs/0.18/oss/guides/expectations/creating_custom_expectations/how_to_add_support_for_the_auto_initializing_framework_to_a_custom_expectation/#modify-the-domain_builder

  • variables: rule共通の変数などを設定する。
    • batch_requestも指定できる。
  • domain_builder: profilingするデータのスコープ範囲を定めるために使用する。
    • ruleに対し1対1である。
    • 単一カラム、複数カラム、テーブルドメインなど
  • parameter_builder: rule固有のパラメータをvariablesなどを利用して設定する。
  • expectation_configuration_builder: rule固有のexpectationsを指定する。
    • ここにvariablesやparameter_builderに設定した変数を指定すること、で柔軟なprofilingが実現できる。

profiler configの実行の仕方#

「profiler_config」にyamlの内容を入れ、他のprofilingと同じようにdata profilingを実行できる。

full_profiler_config_dict: dict = yaml.load(profiler_config)
rule_based_profiler: RuleBasedProfiler = RuleBasedProfiler(
    name=full_profiler_config_dict["name"],
    config_version=full_profiler_config_dict["config_version"],
    rules=full_profiler_config_dict["rules"],
    variables=full_profiler_config_dict["variables"],
    data_context=data_context,
)
batch_request: dict = {
    "datasource_name": "taxi_pandas",
    "data_connector_name": "monthly",
    "data_asset_name": "my_reports",
    "data_connector_query": {
        "index": "-6:-1",
    },
}
result: RuleBasedProfilerResult = rule_based_profiler.run(batch_request=batch_request)

Data Assistantの特長#

WARN：実験的な機能

上記Profilerの設定を、簡易的にしたものがData Asistantである。 複数のBatchRequestのProfilingや結果のplotを、従来のやり方よりも簡単にできるようにすることを目的に当初開発されたようである。 https://docs.greatexpectations.io/docs/0.18/reference/learn/terms/data_assistant/

内部でRule-Based Profilerが使われている。

ざっくり使い方#

以下のように、data_contextへassistantsのaliasを含めrunすると、DataAsistantResultが返ってくる。

assistant = data_context.assistants.<assistantのalias>.run(batch_request)

DataAssistantResultを利用してデータのplotなどを容易にできる。

assistant.plot_expectations_and_metrics()

(appendix)他のプロダクトのprofilingとの比較#

soda#

https://docs.soda.io/soda-cl/profile.html

  • 対応データ

    • テーブル自体(ProfileColumnsResultTable)
    • カラム
      • 文字列
      • datetime
      • 数値

  • GXとの比較

    • カラムの型をメタデータから判別している（https://github.com/sodadata/soda-core/blob/main/soda/core/soda/execution/check/profile_columns_run.py#L45）ので、dataframeには対応していない。
    • プロファイリングしたい内容のカスタマイズなどはできなさそう。

sodaのProfilerは以下にある https://github.com/sodadata/soda-core/tree/7df8e84b5b9bc866653a11134c627624c5fb461b/soda/core/soda/profiling

dbt-profiler#

基本的なカラムの統計値(min,max,data type etc)について精査する。 data frameには対応していない。 https://github.com/data-mie/dbt-profiler/tree/0.8.4/

まとめ#

以上、GreatExpectations(v0)のProfilingとDataAsistantの特長の簡単な説明でした。 前者については、「Rule-Based Profiler」によってdata profilingの柔軟性は増したものの、なかなか設定の複雑さが増しているようにドキュメントやコードを読んでいて感じました。この複雑さが、v1でどのように改善されるのか期待です。
